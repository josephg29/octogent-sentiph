Getting started

Start the app

Run two commands from your project folder. The server manages all agent sessions; the web UI at localhost:5173 is where you work.

Terminal
$ npm install # first time only
$ npm run dev

✓ API server → http://localhost:8787
✓ Web UI → http://localhost:5173
Permissions: By default agents run with --dangerously-skip-permissions — Claude will execute tool calls without prompting you. To require approval on each call, set OCTOGENT_BYPASS_PERMISSIONS=0 before starting.

The orchestrator

Find and open Octoboss

The large red octopus in the center of the canvas is Octoboss. It starts automatically when the server boots — you don't need to create it. Click its node to open its terminal panel on the right side of the screen.

>
  • ✓The dashed outer ring on the Octoboss node shows it is selected / focused.
  • ✓The terminal header shows the tag __octoboss__ — this is its reserved internal ID. Normal workers have short IDs like terminal-a1b2c3.
  • ✓The MCP tools (spawn_terminal and get_terminal_output) are listed in the terminal on startup. No other agent gets these.

4
The key interaction

Type your goal into Octoboss

With the Octoboss terminal open, type your high-level goal at the prompt. Write it as an outcome you want — not a step-by-step script. Octoboss will plan the breakdown and decide which agents to assign.

Write outcomes, not instructions. Say "build the auth flow" — not "first create an agent that does X, then an agent that does Y." Let Octoboss decide how to parallelize the work.
sentiph
octoboss
> Build the auth flow. Create agents for:
the login API, the React form, and tests.
Name each one clearly. Report when all done.
↩ enter to submit
Prompt length limit: Each prompt passed to a worker agent is capped at 8,192 characters. Keep initial instructions focused — workers can ask follow-up questions if needed.
5
Automatic delegation

Workers appear on the canvas

Octoboss calls spawn_terminal once per sub-task. Each call creates a real Claude Code process that appears immediately as a new node on the canvas, connected to Octoboss by a curved edge. New nodes float in and the graph rearranges itself.

sentiph
octoboss
⚡ spawn_terminal
name: "login-api"
→ terminal-a1b2c3

⚡ spawn_terminal
name: "auth-form"
→ terminal-d4e5f6

⚡ spawn_terminal
name: "run-tests"
→ terminal-g7h8i9

● waiting for results...
  • ✦Animated dots on edges — small dots travel along the line between Octoboss and a worker while that worker is actively running. No dots = idle or waiting.
  • ✦Node color — each worker gets a unique color derived from its ID. The color appears on the edge and the node's ring.
  • !Max 9 workers — Octoboss enforces a hard limit of 9 children per parent. It will refuse to spawn a 10th and say so in its terminal.
Rename any terminal by double-clicking its name in the terminal panel header. A text input appears inline. Press Enter to save, Esc to cancel.

7
Completion

Read the final results

When all workers are done, Octoboss collects their outputs and reports a summary back to you in its terminal. Animated edge dots stop moving, indicating workers have gone idle. Click individual worker nodes to inspect their full session output.

octoboss — complete
✓ all 3 workers completed.

summary:
✓ login-api → POST /api/auth/login live, 4 tests passing
✓ auth-form → LoginForm.tsx complete with validation
✓ run-tests → 12 integration tests, all green

files changed:
src/api/auth/login.ts
src/components/LoginForm.tsx
tests/auth.integration.test.ts

>
  • 1Click any worker node on the canvas to open its terminal and read the full session output.
  • 2Delete a finished worker by right-clicking its node → Delete, or clicking ✕ in the terminal column header.
  • !Cascade delete: right-clicking Octoboss → "delete all workers" removes Octoboss and all of its children at once.
  • 3Keep going: type another goal into Octoboss immediately. It spawns a fresh batch for the next task. Previous workers stay on the canvas until you delete them.
Session transcripts are saved automatically to .octogent/state/transcripts/ as JSONL files. Every turn is preserved — you can review any session even after it ends.

Hard limits & tips

9
Max workers per parent
Octoboss refuses to spawn a 10th child. It will say so in the terminal.
8k
Prompt character limit
Each prompt sent to a worker is capped at 8,192 chars.
32
Max concurrent sessions
System-wide cap across all parents and workers combined.
512
Scrollback cap (KB)
Very long sessions have older output trimmed from get_terminal_output reads.
Worker isolation is enforced automatically
Workers do not inherit Octoboss's MCP config — they cannot spawn their own sub-agents. Only Octoboss has spawn_terminal. This prevents runaway costs and keeps you in control. All workers share your project's filesystem by default (shared workspace mode), so they can read each other's files without extra setup. Design parallel tasks to edit different files to avoid last-write-wins conflicts.
Environment variables
# disable auto-accept (Claude will ask before each tool call)
OCTOGENT_BYPASS_PERMISSIONS=0 npm run dev

# state files written automatically at boot
.octogent/state/tentacles.json ← terminal registry
.octogent/state/transcripts/ ← session JSONL logs
.octogent/state/octoboss-mcp-config.json ← MCP tools config
File conflicts: Multiple workers editing the same file simultaneously means last-write wins. Structure parallel tasks so each worker owns different files.
